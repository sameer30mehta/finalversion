# Contributing / Handoff

Workflow for the teammate(s) picking this up.

---

## First time

1. **Clone** the repository and follow `SETUP_FOR_TEAMMATE.md` end to end.
2. Verify the demo runs against the canonical Andheri East case (see
   `DEMO_RUNBOOK.md`).
3. Read `KNOWN_LIMITATIONS.md` so you can speak honestly about scope.

## Branch & PR workflow

- `main` stays demo-ready at all times. Don't push WIP to main.
- Make every change on a feature branch:
  ```powershell
  git checkout -b feat/short-description
  ```
- Open a Pull Request when ready. Self-review the diff before requesting
  another set of eyes.
- For shared / experimental work, prefix branches with `wip/`.

## Things that must NOT be committed

- `.env` and `backend/.env` (they're gitignored — keep them that way).
- `node_modules/` — install per machine.
- `data/*.sqlite`, `data/*.db` — generated by the seed script.
- `.venv/`, `venv/`, `__pycache__/`, `*.pyc`.
- Any uploaded images, generated PDFs, scraped pages, ML model files
  (`*.pkl`, `*.pt`, `*.ckpt`, `*.onnx`).
- Anything under `logs/`, `.cache/`, `.local/`, `.playwright-cache/`.

If you accidentally stage one of these, `git restore --staged <path>` and
add an entry to `.gitignore`.

## Pre-push checklist

Before pushing a branch:

1. **Frontend builds clean**
   ```powershell
   npx vite build
   ```
   Should finish without errors. Warnings about chunk size are fine.

2. **Backend imports + compiles**
   ```powershell
   python -m py_compile backend/main.py backend/llm/ollama_client.py `
     backend/db/repositories.py backend/db/seed_sqlite.py `
     backend/locality_intelligence/pipeline.py
   ```
   Then a quick import smoke:
   ```powershell
   python -c "import backend.main; print('ok')"
   ```

3. **Seed still works** if you've touched anything in `backend/db/` or
   `backend/locality_intelligence/seed.py`:
   ```powershell
   python backend/db/seed_sqlite.py
   ```

4. **No absolute paths**:
   ```powershell
   git grep -nE "D:[\\/]|C:[\\/]|/Users/|/home/" -- ":!node_modules" ":!*.lock"
   ```
   Should return nothing.

5. **No accidentally-staged generated files**:
   ```powershell
   git status
   ```
   Should only show files you actually meant to change.

## Coding rules of thumb

- **Deterministic first.** Never let the LLM or a model affect a number that
  ends up in the audit pack. The model writes narrative; the rules decide
  the math.
- **Cap everything image-derived and locality-derived.** See
  `backend/locality_intelligence/scoring.py` for the cap policy and
  `src/lib/visualEvidenceEngine.js` for the visual caps.
- **Fail soft, badge loudly.** Every external dependency (Ollama, vision
  model, live RSS, official portal) must have a fallback path that is
  labelled in the UI. If a code change can crash the dashboard when the
  dependency is down, it isn't done.
- **Don't introduce machine-specific paths.** Everything reads from env;
  defaults are repo-relative.
- **Don't commit secrets.** Real API keys / credentials never reach the
  repo. Use `.env` and document the variable name in `.env.example`.

## Where to look

| What | Where |
|---|---|
| Deterministic scoring engines | `src/lib/anomalyEngine.js`, `src/lib/historicalReliabilityEngine.js`, `src/lib/valuationEngine.js` |
| Visual evidence rules | `src/lib/visualEvidenceEngine.js` |
| Locality intelligence pipeline | `backend/locality_intelligence/` |
| LLM client + boundary | `backend/llm/ollama_client.py` |
| Backend API endpoints | `backend/main.py` |
| SQLite reference data + seed | `backend/db/` |
| Dashboard sections | `src/components/Dashboard/` |
| Shared UI primitives | `src/components/ui/DashboardPrimitives.jsx` + `src/index.css` |

## Communication

- For non-trivial changes (new endpoint, new dashboard section, scoring rule
  change), align on intent before writing code — open an issue or post in
  the team channel.
- For docs-only changes, push directly to a branch and request review.
- Keep `KNOWN_LIMITATIONS.md` and `DEMO_RUNBOOK.md` truthful as the system
  evolves — they're the documents an outside reviewer will read first.
